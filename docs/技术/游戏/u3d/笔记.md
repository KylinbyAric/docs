# 组件

组件可以理解成一个对象的功能。比如mesh、材质、刚体、灯光等等。对象之间的区别就在于挂载的组件不一样。特殊的一点是所有的对象都有transform(位置组件)，因为这是物体存在于空间的标识。

### 🌀 行为微调三件套：弹簧、电机与角度限制

你可以通过组合使用这三个功能，获得想要的细腻物理效果：

* **开启 `Use Limits` 限制范围**：为关节设定转动角度的上下限，比如只让门能在0°到90°之间开合。对于角色肢体关节，必须开启“限制”以防止不自然的弯曲。
* **开启 `Use Spring` 实现自动归位**：开启后，物体会自动转向目标角度，模拟“自动归位”效果，比如让一扇推开的门自己缓缓关上。
* **开启 `Use Motor` 主动驱动旋转**：开启后，物体会按指定速度和力量主动旋转，很适合需要自动运行的机关。

### ⚠️ 常见误区与避坑指南

#### 🎡 误区一：旋转轴（Axis）设置错误，导致物体乱转、抖动甚至卡死

这是最常见的错误之一，即旋转轴（Axis）的方向设置不当。例如，**如果你想要一个平开门绕Y轴转动，但你设置的是绕X轴旋转，物理模拟就会失效**。

* **错误表现**：关节无法按预期绕指定轴旋转，出现奇怪抖动。
* **解决方案**：在 `Inspector` 面板的 `Axis` 属性区域，直接点击并拖动坐标轴箭头，或输入数值。**特别注意：模型导入和旋转角度会影响局部坐标轴方向。**

#### 🛠️ 误区二：刚体与碰撞体配置不当引发的物理冲突

当关节连接的物体与玩家角色等碰撞时，经常会出现剧烈抖动，甚至整个结构散架。这往往是因为：连接的刚体质量相差太大，导致解算困难；或是一个刚体被玩家顶住，而它的连接点又被其他物体卡住，两者互相矛盾，物理引擎无法解算，于是抖动。

**解决方案：**

* **调整质量比**：尽量让关节连接的刚体质量相近。如果必须相差悬殊，可尝试启用 `Enable Preprocessing` 并适当调节 `Mass Scale` 属性。
* **检查层级**：使用**刚体组件的Sleeping功能**与**Layer Collision Matrix（层碰撞矩阵）**，禁用无关碰撞。
* **调整参数**：对于核心关节，微调 **Solver Iterations** 和 **Solver Velocity Iterations**，可以提升解算精度和稳定性。

#### 🌲 误区三：链条过长或受到高速冲击时，关节链接处出现分离、拉伸或断裂

当铰链连接的链条过长或被高速挥舞时，链接处容易出现分离、拉伸或断裂。这是因为默认的Break Force/Break Torque参数设置不当，或者链条各段的质量与受力关系不匹配。可以调整刚体的阻力（Drag）角阻力（Angular Drag）增加稳定性；同时，为应对物理误差，尝试使用**DistanceJoint2D**作为辅助。

### 🚀 性能优化建议

* **减少关节数量，必要时物理烘焙**：物理运算是性能开销的一部分，应使用 `GameObject` 的 Active/Deactive 或脚本控制关节的休眠来节省性能。
* **限制参与碰撞检测的物体**：善用 `Layer Collision Matrix`，**避免关节上的物体与不需要交互的物体（如场景装饰）发生碰撞**，减少不必要的物理计算。
* **控制迭代次数**：仅在必要时，为关键物体适当提高 `Solver Iterations`，避免全局设置过高导致性能消耗过大。
* **使用层级结构并降低迭代**：合并网格并合理设置父子层级，可以减少物理计算量。对表现效果影响不大的关节，适当调低刚体组件的迭代次数（Solver Iterations）。

### ⚙️ 进阶：与其它关节的比较


| 关节类型                            | 核心特性                                                                       | 典型应用                                 |
| ----------------------------------- | ------------------------------------------------------------------------------ | ---------------------------------------- |
| **Hinge Joint (铰链关节)**          | 模拟单轴旋转，如门轴或钟摆，是最简单直观的旋转关节                             | 普通门、链条、秋千、钟摆                 |
| **Fixed Joint (固定关节)**          | 将两个物体刚性连接，如同焊接在一起，使其保持固定的相对位置                     | 组合碎片、无需相对运动的机械零件         |
| **Spring Joint (弹簧关节)**         | 在两个物体间模拟弹簧的弹性连接，产生推拉效果                                   | 弹性绳索、悬挂系统、有缓冲效果的连接     |
| **Character Joint (角色关节)**      | 模拟球窝关节，常用于布娃娃系统，限制在锥形区域内旋转                           | 角色四肢、头部与身体的连接               |
| **Configurable Joint (可配置关节)** | 提供最高自由度的关节，可精确控制各运动轴的驱动与限制，是其他所有关节的“母体” | 复杂的机械臂、仿生关节、精细控制的布娃娃 |

## 刚体

### ⚙️ 核心参数功能速查

这是 Rigidbody 组件最核心的参数及其主要功能：


| 参数                                   | 功能                                                                             | 典型值与说明                                                                                   |
| -------------------------------------- | -------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **Mass (质量)**                        | 物体的惯性和与其他物体碰撞时的动量。质量大的物体会推动质量小的物体。             | 默认 1.0。建议不同物体的质量比不超过**10:1**，避免物理模拟不稳定。                             |
| **Drag (阻力)**                        | 线性运动的阻力，用于模拟空气阻力或摩擦力。值越大，物体减速越快。                 | 角色移动建议设为**1-5**，避免“滑冰”感。                                                      |
| **Angular Drag (角阻力)**              | 旋转运动的阻力，值越大，旋转停得越快。                                           | 默认 0.05，保持合理范围（如**0.05-1**）。                                                      |
| **Use Gravity (使用重力)**             | 控制物体是否受重力影响。默认启用。                                               | 通常保持启用，除非需要漂浮等特殊效果。                                                         |
| **Is Kinematic (是运动学)**            | 物体**不受物理引擎控制**，但仍能影响其他物理物体。常用于移动平台或可搬运的道具。 | 启用时，需通过脚本修改 Transform 或 Rigidbody.MovePosition 来控制它。                          |
| **Interpolate (插值)**                 | 平滑物理位置在渲染帧之间的显示，防止物体运动时出现抖动或跳跃。                   | 对跟随摄像机的玩家角色很有用。                                                                 |
| **Collision Detection (碰撞检测模式)** | 决定物理引擎如何检测碰撞，是**精度与性能平衡**的关键。                           | 绝大多数物体保持默认的**Discrete**，快速移动的物体需设为 **Continuous / Continuous Dynamic**。 |
| **Constraints (约束)**                 | 锁定物体的位置或旋转轴，防止物理模拟导致其在某些方向上移动或旋转。               | 常用的是冻结旋转轴的 Z 与 X 轴。                                                               |

> **关于质心与惯性张量**：这两个是更高级的属性。`Mass`、`Drag`、`Angular Drag` 是**最常用且对物理行为影响显著**的核心属性。**质心**影响物体的平衡（例如，一个高重心的物体更容易倾倒），而**惯性张量**决定物体旋转的难易程度。手动调整这些能制作出更真实的物理效果，但非必要不建议修改。

### 📜 常见应用场景

* **3D/2D 平台跳跃 (角色控制)**：通过 `Rigidbody.AddForce()` 施加瞬间跳跃力；并为地面、墙壁等环境物体添加碰撞体。
* **爆炸冲击波 (物理反馈)**：计算爆炸范围内的所有物体，根据距离和方向，用 `Rigidbody.AddExplosionForce()` 模拟冲击力。
* **车辆竞速 (载具模拟)**：在四个轮子的位置施加力，根据车轮转向调整力的方向，模拟车辆的前进、后退和转向。
* **机关解谜 (物体交互)**：利用投掷物触碰压力板触发机关，关键在于为机关部分添加碰撞触发器（`Is Trigger`）。

### ⚡ 核心代码 API (C#)

在脚本中控制刚体的核心方法如下，注意物理相关的代码应放在 `FixedUpdate` 函数中。


| 核心方法                              | 关键参数说明                                                                                                                                      | 示例代码                                                                               |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **`AddForce` (施加力)**               | `ForceMode`：四种施力模式<br/>- `Force`：持续力（默认）<br/>- `Impulse`：瞬间冲量<br/>- `Acceleration`：加速度<br/>- `VelocityChange`：直接改速度 | `rb.AddForce(Vector3.forward * 10f, ForceMode.Impulse);`                               |
| **`AddTorque` (施加扭矩)**            | 施加旋转力，用法与`AddForce`类似。                                                                                                                | `rb.AddTorque(Vector3.up * 100f);`                                                     |
| **`MovePosition` (移动位置)**         | 让运动学刚体 (`isKinematic = true`) 平滑移动。                                                                                                    | `rb.MovePosition(rb.position + velocity * Time.fixedDeltaTime);`                       |
| **`MoveRotation` (移动旋转)**         | 让运动学刚体平滑旋转。                                                                                                                            | `rb.MoveRotation(rotation * Quaternion.Euler(0, turnSpeed * Time.fixedDeltaTime, 0));` |
| **`AddForceAtPosition` (施加位置力)** | 在世界坐标的特定点施加力，常用于制造扭矩。                                                                                                        | `rb.AddForceAtPosition(force, point, ForceMode.Impulse);`                              |
| **`velocity` / `angularVelocity`**    | 直接读取或设置刚体的线速度和角速度（矢量）。                                                                                                      | `rb.velocity = Vector3.forward * speed;`                                               |
| **`Sleep` / `WakeUp` (睡眠/唤醒)**    | 手动控制刚体进入/退出“睡眠”状态，是**重要的性能优化手段**。                                                                                     | `rb.Sleep();` 或 `rb.WakeUp();`                                                        |
| **`isKinematic` (是否运动学)**        | 控制刚体是否受物理模拟影响。                                                                                                                      | `rb.isKinematic = true;`                                                               |

### ⚠️ 常见误区、问题及解决方案

* **❌ 误区1：通过 Transform 直接移动物体**：直接修改 `transform.position` 会绕过物理引擎，导致穿墙、碰撞错误或动画撕裂。**✅ 正确做法**：对动态物体使用 `AddForce`，对运动学物体使用 `MovePosition`。
* **❌ 误区2：计算力与速度时，未乘以 `Time.deltaTime`**：物理施力应在 `FixedUpdate` 中进行，此时已内置了 `Time.fixedDeltaTime` 的时间步长计算，因此**施加力时不需要再额外乘以 `Time.deltaTime`**。
* **❌ 误区3：将 Collider 和 Rigidbody 都设为触发器 (Trigger)**：如果一个物体**同时勾选 `IsTrigger` 并附加了刚体**，它仍会参与物理模拟，可能导致不可预知的运动和穿透。**✅ 最佳实践**：触发器通常只用于检测，若必须移动它，请使用运动学刚体（`isKinematic = true`）。
* **❌ 误区4：父子级碰撞检测失效**：将一个有刚体的物体设为另一个有刚体的物体的子物体时，`OnCollisionEnter` 等方法可能无法正常触发。**✅ 解决方案**：尽量将刚体放在最高层级的物体上，避免复杂的嵌套刚体结构。
* **⚠️ 问题1：物体下落/运动缓慢**：这可能源于模型的比例不当。Unity 物理系统默认 1 单位 = 1 米，如果你的小车模型长 50 个单位，在物理引擎眼中就是一个 50 米长的庞然大物，运动看起来自然会非常缓慢。**解决方案**：在建模或导入时，确保物体的尺寸与现实世界比例相近。
* **⚠️ 问题2：物体无限弹跳**：当一个物体的某些位置轴被冻结（`Freeze Position`），碰撞时产生的力无法被完全吸收，就会造成持续弹跳。**解决方案**：尝试解冻该物体的 `Freeze Position` 设置。
* **⚠️ 问题3：刚体穿透 (Tunneling)**：当一个物体移动速度过快时，可能会在两个物理帧之间直接穿过另一个物体，从而不发生碰撞。**解决方案**：为子弹等快速移动的物体将碰撞检测模式设为 `Continuous` 或 `Continuous Dynamic`。

### 🚀 性能优化与实践

* **启用“刚体休眠”**：当物体的速度低于阈值时，Unity 会自动让它进入“睡眠”状态。其下落的视觉速度会相当缓慢——物理引擎会暂停对它的计算以节省 CPU 性能。
* **选择合适的碰撞检测模式**：默认的 **Discrete** 性能最佳。仅在必要处（如子弹）使用开销更大的**连续碰撞检测模式**。
* **使用 Solver Iterations (求解迭代次数)**：如果发现物理堆叠或连接过于不稳定，可针对性增加该刚体的 `solverIterations` 属性值（每次物理步进的解算次数），来提升当前物体的稳定性与准确性，这对单个复杂物体比全局调高更高效。
* **善用层碰撞矩阵 (Layer Collision Matrix)**：在 `Project Settings > Physics` 中，通过矩阵**禁用不相关层之间的碰撞检测**，能有效减少无效计算。
* **其他全局优化**：在 Project Settings → Physics 中，可以通过调整 `Fixed Timestep` 等全局参数来平衡物理精度和CPU负载。

### 碰撞模式

#### 一、三种模式对比


| 模式                                 | 原理                                                                                                       | 适用场景                                                     | 性能开销             | 能否防止高速穿透                                |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ | -------------------- | ----------------------------------------------- |
| **Discrete**（离散）                 | 每帧只检测物体从**上一帧位置**到**当前帧位置**的终点处是否有碰撞。                                         | 绝大多数普通物体（角色、静态障碍物、低速移动物）。           | **最低**（默认模式） | ❌ 不能                                         |
| **Continuous**（连续）               | 对**射线**和**连续检测物体**，会扫描从上一帧位置到当前帧位置的整个运动路径。                               | 高速移动且需要**被其他连续物体撞击**的物体（如子弹、飞镖）。 | **中等**             | ✅ 能防止被其他连续物体穿透，但对静态碰撞体无效 |
| **Continuous Dynamic**（动态连续）   | 更严格的连续检测，既能扫描自身运动路径，也能**与所有 Continuous 和 Continuous Dynamic 物体**进行扫描碰撞。 | 需要**被高速子弹击中**的物体（如易碎花瓶、玩家角色）。       | **最高**             | ✅ 能防止与任何连续检测物体的穿透               |
| **Continuous Speculative**（推测式） | 基于推测算法，在运动发生前预测潜在的碰撞并提前处理，开销小于完全的扫描检测。                               | 移动端或性能敏感场景下需要防穿透的物体。                     | **中等偏低**         | ✅ 能防止穿透（精度略低于 Continuous）          |

> **注意**：`Continuous Speculative` 是 Unity 2017.3 之后加入的模式，并非所有旧文档都会提到。

#### 二、典型错误与正确配置


| 错误认知                                              | 正确解释                                                                                                                 |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 把**所有物体**都设为 Continuous 或 Continuous Dynamic | 会严重降低物理性能。只为**必要的高速物体**开启。                                                                         |
| 为静态碰撞体（无 Rigidbody）设置连续检测              | 连续检测只对**刚体**有效，静态碰撞体永远使用离散检测。                                                                   |
| 认为 Continuous 能防止与**静态网格**的穿透            | Continuous**仅与带有连续检测的刚体**进行扫描。要防止与静态碰撞体的穿透，需改用 Continuous Speculative 或增大碰撞体厚度。 |
| 在 2D 物理中混淆                                      | 2D 刚体（Rigidbody2D）的连续检测模式是`Continuous` 和 `ContinuousDynamic`，用法与 3D 类似。                              |

## 灯光

### 🏮 Unity 光源详解 (Light Component)

Unity的`Light`组件是我们工作的核心。下面我们通过一个表格来总览其主要参数，然后再逐一细说。

#### ⚙️ 核心参数一览

下表汇总了`Light`组件的关键参数及其核心作用：


| 参数分类           | 参数名                 | 核心作用                        | 关键提示                         |
| ------------------ | ---------------------- | ------------------------------- | -------------------------------- |
| **基础**           | `Type`                 | 定义光源的形状和发光方式        | 影响最根本的性能和效果           |
|                    | `Color`                | 设置光源颜色                    | 影响场景色调                     |
|                    | `Intensity`            | 控制亮度                        | 默认值：方向光0.5，其他1.0       |
|                    | `Range`                | 控制光照范围                    | 仅点光源和聚光灯适用             |
|                    | `Spot Angle`           | 控制聚光灯光锥角度              | 仅聚光灯适用                     |
| **模式 (Mode)**    | `Mode`                 | 决定是实时计算还是预先烘焙      | **性能的关键开关**               |
| **阴影 (Shadows)** | `Shadow Type`          | 决定是否投射阴影                | 柔和阴影比硬阴影更耗性能         |
|                    | `Strength`             | 控制阴影的暗度                  | 0为无阴影，1为全黑               |
|                    | `Resolution`           | 控制阴影贴图分辨率              | 更高的分辨率性能消耗更大         |
|                    | `Bias` / `Normal Bias` | 修正阴影渲染时的瑕疵            | 过高会导致阴影与物体分离         |
| **高级**           | `Cookie`               | 在灯光前增加图案遮罩            | 模拟纹理投影，如窗户或树叶的影子 |
|                    | `Culling Mask`         | 控制灯光影响哪些层(Layer)的对象 | 通过剔除无关物体来优化性能       |
|                    | `Render Mode`          | 控制Unity如何渲染该光源         | 覆盖自动分类，手动管理性能       |

接下来，我们对其中最关键的**类型**和**模式**进行深入解读。

#### 🔦 四大光源类型 (Type)

首先，你需要根据场景需求选择合适的光源类型，每种都有其独特的用途和性能消耗。


| 类型                     | 简介                                       | 性能与最佳实践                                                                                                 |
| ------------------------ | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| **方向光 (Directional)** | 模拟太阳，由方向决定，不受位置影响         | **性能最佳**，一个新场景通常只有一个方向光，但注意旋转它会影响天空盒。                                         |
| **点光源 (Point)**       | 从一点向所有方向发光，模拟灯泡、烛火       | **性能中等，但阴影消耗巨大**。因为它投射阴影需要6张阴影贴图，开销远超聚光灯的1张。适合精确控制范围和使用烘焙。 |
| **聚光灯 (Spot)**        | 从一点向锥形区域发光，模拟手电筒、舞台追光 | **性能优于点光源**，其锥形范围便于进行视锥剔除。常用于室内，也是最灵活的补光工具之一。                         |
| **面光源 (Area)**        | 从一个矩形或圆盘面均匀发光                 | **运行时不支持，仅用于烘焙**。其光照柔和，非常适合模拟大面积光源，如发光的窗户或柔光箱。                       |

#### 💡 三大光照模式 (Mode)

选择了类型后，再通过`Mode`来决定是实时计算性能开销，还是预先“烘焙”来换取性能。


| 模式                | 行为                                                                   | 优缺点                                                                                                                        |
| ------------------- | ---------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **实时 (Realtime)** | Unity每帧都会计算光源的亮度和阴影。                                    | ✅ 优点：完全动态，光源可移动，颜色强度可随代码改变。<br/>❌ **缺点：性能开销极高**，动态阴影是渲染中最昂贵的特性之一。       |
| **烘焙 (Baked)**    | 在编辑器中预先计算好光照效果，生成光照贴图。运行时使用贴图，不再计算。 | ✅ 优点：性能消耗极低，渲染效果平滑，可表现复杂的间接光照（颜色反弹）。<br/>❌ 缺点：所有效果都是**静态**的，运行时无法改变。 |
| **混合 (Mixed)**    | 将光影拆分为直接光和间接光，提供“部分动态”方案。                     | ✅ 优点：是表现力与性能的折中选择。例如，动态角色可以接收从静态环境光来的光影效果。<br/>❌ 缺点：配置复杂，且仍有性能成本。   |

### 📜 Lighting 窗口：控制全局光与氛围

除了每个光源的单独参数，`Lighting`窗口负责控制场景的全局光照、环境光和阴影系统，是定调画面风格的核心。

* **环境光 (Environment Lighting)**：定义了没有光源直射区域的默认亮度和颜色。`Source`参数让你可以选择将天空盒、纯色或自定义渐变作为场景的环境照明基础。
* **全局光照 (Global Illumination, GI)**：模拟光在物体表面间反弹的逼真效果。`Lighting`窗口的`Realtime Global Illumination`和`Baked Global Illumination`是其两大技术支柱，决定了如何为场景计算和生成丰富的光影层次。
* **其他设置**：包括控制场景雾效的`Fog`，以及控制阴影显示距离和质量的`Shadow Distance`、`Shadow Cascades`等。

### ⚠️ 常见陷阱 (Pitfalls)

1. **滥用点光源阴影**：实时点光源投射阴影的性能消耗是聚光灯的6倍，应谨慎用于运行时。
2. **误解实时渲染**：应使用最合适的（例如，用面光源营造柔和光影），而不是在性能上“挥霍”。
3. **阴影偏差 (Bias) 问题**：阴影断裂或“漏光”，通常是`Bias`值过高，调低即可。
4. **超出阴影距离**：远处的物体阴影突然消失，是`Shadow Distance`设置过小，调大即可。
5. **忽略剔除设置**：灯光应只照射必要物体。通过`Culling Mask`减少不必要的计算。

### 🚀 性能优化实践 (Optimization)

1. **区分 "逐像素" 与 "逐顶点"**：在质量设置中，优先用最昂贵的**逐像素** 光照服务最核心的物体（如角色），次要的用效率更高的**逐顶点**或**球谐函数 (SH)** 光照处理，这是效率分层的关键。
2. **使用渲染模式 (Render Mode)**：为关键光源标记`Important`以强制其进入最高优先级队列，无关紧要的光源标记为`Not Important`以节省性能。
3. **适度使用阴影**：能不投射阴影的光源就不要投射。并且，只对场景中最重要的光源开启最高质量的阴影贴图。
4. **减少实时光源与阴影**：在移动平台，有每个对象的实时光源数量限制（OpenGL ES 2.0 限制为 4 个），应极力避免视觉冲突。

# 摄像机

### 🔧 一、核心属性与参数详解

摄像机的 Inspector 面板中，部分核心属性的作用和优化建议如下：


| 属性                                          | 作用                                                                                                                                                                 | 优化建议                                                                                                                                                    |
| --------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Clear Flags**                               | 决定当前摄像机如何清除屏幕上的现有内容。每个摄像机在渲染时都会先清空屏幕（如填充纯色、天空盒等），以绘制自己的画面。<br />个人理解：就是摄像机背景，不会出现在场景里 | 当使用多个摄像机叠加UI时，可以设置为`Depth only` 清除深度但不覆盖背景，以保证UI渲染在正确的前景。                                                           |
| **Projection** (`Perspective`/`Orthographic`) | 透视模式模拟真实人眼，有近大远小的效果；正交模式则无视距离，常用于 2D 游戏或**小地图**。                                                                             | 正交模式下的雾效渲染可能不符合预期，因为它会均匀应用在屏幕上。2D 游戏建议使用正交模式。                                                                     |
| **Field of View / Size**                      | **视场角（FOV）** 决定了透视摄像机的视野广度，数值越大，能看到的区域越宽，但边缘的畸变也越严重；**大小（Size）** 则决定了正交摄像机的视野高度。                      | FOV 数值过大会导致画面畸变，过小则视野狭窄。通常设置`60-90` 是标准的舒适范围。                                                                              |
| **Clipping Planes**                           | 定义了摄像机开始和停止渲染的距离范围。超出`Far` 平面的物体会直接被“裁剪”（Cull），不再渲染。<br />个人理解：远面和近面(靠近摄像机)之间的物体才会被渲染             | **远平面（Far）** 的设置对性能影响巨大，`1000` 通常已足够覆盖大多数3D场景。                                                                                 |
| **Viewport Rect**                             | 定义了摄像机画面在屏幕上的位置和大小。                                                                                                                               | 常用于实现**分屏游戏**：为不同玩家单独配置摄像机，并分别设置视口区域。                                                                                      |
| **Depth**                                     | 数字越大，摄像机绘制时越靠前，画面会覆盖 Depth 值较小的摄像机。可以理解为主页面的优先级，数字越大越优先展示                                                          | 配合`Viewport Rect` 使用，可以非常方便地实现 **小地图** 或 UI 效果。                                                                                        |
| **Rendering Path**                            | 决定 Unity 渲染光照和阴影的方式，对画质和性能影响极大。                                                                                                              | **前向渲染（Forward）** 兼容性好但对光源开销大，适合移动端；**延迟渲染（Deferred）** 支持大量实时光源，但需要硬件支持且抗锯齿处理复杂，适用于高端PC和主机。 |
| **Target Texture**                            | 可选的高级特性。可将摄像机的画面输出到一张纹理上，而非直接显示在屏幕。                                                                                               | 经典应用是制作**监控摄像头、后视镜或游戏内的“照片”功能**。                                                                                                |
| **Culling Mask**                              | 允许你指定摄像机只渲染哪些**层（Layer）** 上的物体。                                                                                                                 | 通过勾选无关的层来忽略渲染（如UI层、特效层），是优化性能的利器。                                                                                            |
| **Occlusion Culling**                         | 一种高级优化功能。它**不渲染那些完全被其他物体遮挡** 的物体，即使它们位于摄像机的视锥体内。<br /><br />个人理解就是不渲染指定图层的物体。                            | 非常适合**室内场景**，可以有效减少复杂环境下的渲染压力。                                                                                                    |

### 二、四种渲染模式详解（Render Mode）

`Render Mode` 决定了视频最终显示在什么位置，是 Video Player 最核心的设置之一。


| 渲染模式              | 工作原理                                              | 适用场景                                                      | 性能     | 注意事项                                     |
| --------------------- | ----------------------------------------------------- | ------------------------------------------------------------- | -------- | -------------------------------------------- |
| **Camera Far Plane**  | 视频渲染到摄像机远平面上，出现在所有场景物体**背后**  | 动态背景、远景天空盒、环境氛围                                | **低**   | 需要指定目标摄像机                           |
| **Camera Near Plane** | 视频渲染到摄像机近平面上，出现在所有场景物体**前方**  | 全屏过场动画、UI 叠加特效、视频广告                           | **低**   | 会遮挡所有场景内容，适合过场                 |
| **Render Texture**    | 视频渲染到 Render Texture，可赋值给任意材质的纹理属性 | UI 界面（RawImage）、3D 物体表面（电视/广告牌）、摄像头监视器 | **中**   | 灵活性最高，需要额外创建 Render Texture 资源 |
| **Material Override** | 直接覆盖游戏对象材质上的指定纹理属性                  | 3D 物体表面直接播放视频（无需 Render Texture）                | **中高** | 物体必须有 Renderer 组件                     |

**选择建议**：背景/过场用 Camera Plane，UI 用 Render Texture + RawImage，3D 表面可用两者均可。在性能敏感场景（如移动端），Camera Plane 模式对 GPU 的压力最小。

---

### 三、视频格式与导入设置

#### 1. 支持的编解码器与格式


| 平台                           | 推荐编解码器            | 容器格式               | 说明                                  |
| ------------------------------ | ----------------------- | ---------------------- | ------------------------------------- |
| **Windows / macOS / 主流平台** | **H.264**               | `.mp4`, `.mov`, `.m4v` | 硬件加速支持最佳，跨平台兼容性最好    |
| **Linux 编辑器**               | **VP8** + Vorbis 音频   | `.webm`                | Linux 不支持 H.264                    |
| **WebGL**                      | H.264 / VP8             | `.mp4` / `.webm`       | Video Clip 方式不支持，需用 URL       |
| **透明视频**                   | VP8 / Apple ProRes 4444 | `.webm` / `.mov`       | 仅这两种编码原生支持每像素 Alpha 通道 |

Unity 支持的常见视频容器格式包括：`.avi`, `.mov`, `.mp4`, `.mpg`, `.webm`, `.wmv` 等。

#### 2. Video Clip Importer 关键设置

导入视频到 Unity 后，在 Inspector 中可以设置以下参数：


| 设置项                 | 功能                                                                       |
| ---------------------- | -------------------------------------------------------------------------- |
| `sRGB (Color Texture)` | 默认开启，适用于普通视频颜色空间；非色彩用途（如数据视频）可关闭以提升性能 |
| `Transcode`            | 勾选后 Unity 会将视频转码为目标平台兼容格式                                |
| `Codec`                | 转码时使用的视频编码（H.264 推荐）                                         |
| `Bitrate Mode`         | 控制视频码率，码率越高画质越好但文件更大                                   |
| `Spatial Quality`      | 是否在转码时压缩图像尺寸（节省空间但会降低画质）                           |

### 四、播放透明视频（Alpha 通道）

Unity 支持两种透明视频方案：


| 方案             | 原理                                                         | 实现方式                                                                          | 适用场景                            |
| ---------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------- | ----------------------------------- |
| **全局 Alpha**   | 整个视频统一透明度，Render Mode 必须是 Camera Near/Far Plane | Video Player 组件中调整`Alpha` 值（0\~1）                                         | 简单的淡入淡出效果，如渐显/渐隐过场 |
| **每像素 Alpha** | 视频不同区域有不同透明度                                     | 需使用支持 Alpha 的视频编码：**VP8 (WebM)** 或 **Apple ProRes 4444 (macOS Only)** | UI 特效、角色特效、粒子叠加等       |

**透明视频的常见质量陷阱**：

* H.264 不支持 Alpha 通道，用普通解码器强行解码会导致边缘模糊或锯齿；
* 若视频分辨率与 Render Texture 不匹配，缩放会导致画质损失，建议两者保持一致；
* VP8/VP9 解码比 H.264 消耗更大，移动端尤其明显。

### 五、应用场景速览


| 场景                  | 推荐 Render Mode          | 关键要点                                                     |
| --------------------- | ------------------------- | ------------------------------------------------------------ |
| **游戏开场动画/过场** | Camera Near Plane         | 全屏播放，不受场景物体遮挡，Loop 关闭                        |
| **动态远景背景**      | Camera Far Plane          | 置于摄像机最远处，营造环境氛围                               |
| **电视/电脑屏幕播放** | Render Texture            | 创建 Render Texture 赋给屏幕材质，搭配 RawImage 实现 UI 播放 |
| **可交互广告牌**      | Material Override         | 直接覆盖物体材质纹理，适合建筑屏幕等                         |
| **UI 界面视频**       | Render Texture + RawImage | UI 上需要视频组件时必用                                      |
| **360° 全景视频**    | Material Override         | 分配给 Skybox 或 Sphere 材质                                 |
| **特效粒子视频**      | 每像素 Alpha + VP8        | 透明视频实现粒子特效、魔法效果等                             |

### 六、常见误区与踩坑点

#### ⚠️ 误区 1：画面残留 / 画面闪烁（切换视频时出现上一条视频的最后一帧）

**现象**：切换视频时屏幕闪现上一个视频的最后一帧。原因是切换过程中新视频需要 1-2 帧加载，而旧视频的最后画面仍留在 Render Texture 上。

**解决方案**：可在切换新视频前先调用 `Stop()` 停止旧播放，或用脚本将 Render Texture 清除为纯色。对于资源充足、切换不频繁的场景，新建 Render Texture 也能彻底避免残留。

#### ⚠️ 误区 2：网络视频无法播放

* Android 平台的 URL 需要加上 `"http://"` 前缀；
* iOS 需确保 App Transport Security (ATS) 允许非 HTTPS 连接（或在 info.plist 中配置例外）。

#### ⚠️ 误区 3：移动端卡顿/发热严重

* 视频分辨率过大（如 4K）而设备无法硬件解码；
* VP8/VP9 编码的视频在部分平台上被迫使用软件解码，CPU 负载极高；
* 直播流或远程 URL 频繁缓冲。

#### ⚠️ 误区 4：透明视频边缘发黑/锯齿

* 使用了不支持 Alpha 的视频格式（如普通 MP4），应改用 VP8 (WebM)；
* 视频分辨率与 Render Texture 不一致导致采样失真，建议两者尺寸匹配；
* 渲染顺序（Render Queue）或混合模式设置不当，导致半透区域与背景叠加错误。

#### ⚠️ 误区 5：视频声音缺失

* 视频文件中音频轨道未被正确导入：检查 Video Clip Importer 中 `Import Audio` 是否勾选；
* 场景中没有 `AudioSource` 组件：Video Player 需要依赖一个 AudioSource 才能输出音频，可以在挂载 Video Player 的游戏对象上同时挂载 AudioSource，然后将 Video Player 的 `Audio Output Mode` 设置为 `AudioSource` 并指向它。

#### ⚠️ 误区 6：在 WebGL 平台视频无法播放

* `Video Clip` 方式在 WebGL 上不被支持，必须改用 `URL` 方式，将视频放置在服务器并通过 http:// 地址访问。

### 七、性能优化方案

#### 1. 编码与分辨率控制


| 优化手段                               | 效果                                     |
| -------------------------------------- | ---------------------------------------- |
| 使用**H.264** 编码替代 VP8/VP9         | 硬件解码支持好，CPU/GPU 负载更低         |
| 降低视频分辨率和帧率                   | 移动端尤其显著，推荐 ≤1080p，帧率 30fps |
| 启用 Video Clip Importer 的`Transcode` | 自动优化为目标平台的格式                 |
| 控制码率（Bitrate）                    | 码率过高浪费性能，过低影响画质           |

#### 2. 加载与资源管理


| 优化手段                      | 效果                                             |
| ----------------------------- | ------------------------------------------------ |
| **预加载/异步加载**           | 提前加载视频资源，避免播放时触发 I/O 延迟        |
| **使用 StreamingAssets**      | 外部存储大视频，避免嵌入构建包增大体积           |
| **及时释放 VideoPlayer 实例** | 调用`Stop()` 并销毁 Render Texture，防止内存泄漏 |
| **启用视频缓存**              | 网络流媒体可缓存数据减少重复读取                 |

#### 3. 多 Video Player 管理

当场景中需要**同时播放多个视频**时，需要更精细地平衡性能与效果：

* 如果同一时间只需要显示一个视频，应采用 **单实例复用** 策略：只维护一个 Video Player 对象，播放新视频前先 `Stop`、清除资源、更换源后再 `Play`，避免多个解码器同时占用资源；
* 如果**必须同时播放多个视频**，优先将次要视频设为较低分辨率，并尽量使用 `Camera Far/Near Plane` 模式（开销低于 Render Texture），严格控制同时活跃的播放器数量。

#### 4. 移动端进阶优化

移动平台需额外关注：

* **Surface 生命周期陷阱**：当 Activity 被系统回收时（如切换 App），若 VideoPlayer 未显式调用 `Stop()` 并置空 `targetTexture`，将导致 OpenGL 上下文异常与黑屏；
* **硬解降级**：当硬件解码失败时（如特定 SoC 不兼容），自动降级为软件解码（使用 UnityWebRequest + 备用方案）；
* **内存水位监控**：监控 VideoPlayer 纹理内存占用（Android 建议 ≤32MB），超阈值时触发纹理回收。

# 脚本

## 生命周期

Awake：初始化时会调用，**只调用一次**。主要是进行对相见的初始化、单例等。
OnEnable：激活的时候会调用。每次对象变为激活时调用。
Start:首次update前会调用，只调用一次。
FixedUpdate：固定时间间隔调用，默认是0.02s。类似定时器
Update: 每一帧都会调用。常用的游戏逻辑处理。
LateUpdate：每一帧在所有脚本的Update方法调用之后调用。类似钩子函数。如摄像机的跟随等。
OnDisable:附着对象变为非激活状态时调用，主要用于注销、保存等操作。
OnDestroy:附着对象被销毁时调用，主要是进行最终清理。

### 激活和销毁的区别


| 维度             | 激活/禁用 (Active/Inactive)                             | 销毁 (Destroy)                              |
| :--------------- | :------------------------------------------------------ | :------------------------------------------ |
| **对象是否存在** | 对象**仍然存在于场景**中，只是被标记为不可见/不参与逻辑 | 对象**被从场景中彻底移除**，引用变为 `null` |
| **内存占用**     | **仍占用内存**（组件、数据都保留）                      | **释放内存**（除非有其他引用阻止垃圾回收）  |
| **能否恢复**     | **可以**，通过 `SetActive(true)` 立即恢复               | **无法恢复**，需重新实例化或从预制体生成    |
| **触发生命周期** | `OnEnable` / `OnDisable`                                | `OnDestroy`                                 |
| **对子对象影响** | 递归影响所有子对象                                      | 递归销毁所有子对象                          |
| **典型用途**     | 临时隐藏物体（如 UI 面板、子弹对象池）、暂停逻辑        | 永久删除物体（如敌人死亡、场景切换清理）    |

激活时
对象仍然存在，只是不可见。
仍占用内存
临时隐藏物体，UI面板、暂停等

销毁时
彻底移出场景
不再占用内存
敌人死亡/场景切换清理

## 执行顺序


| 维度                    | 规则 / 说明                                                                                                                               |
| :---------------------- | :---------------------------------------------------------------------------------------------------------------------------------------- |
| **整体阶段顺序**        | **固定**：`Awake` → `OnEnable` → `Start` → `Update` → `LateUpdate` → …（所有脚本先完成同一阶段，再进入下一阶段）                    |
| **同一阶段内（默认）**  | **不确定**，由 Unity 内部决定，无法预测谁先执行                                                                                           |
| **同一阶段内（可控）**  | 依**执行优先级数值从小到大**（升序）执行，数字越小越先执行                                                                                |
| **同一脚本的不同实例**  | **无法单独控制**顺序，只能控制不同脚本类之间的顺序                                                                                        |
| **设置方法① - 编辑器** | `Edit` → `Project Settings` → `Script Execution Order`，添加脚本并调整序号（数字越小越先）                                              |
| **设置方法② - 代码**   | 在脚本类上添加`[DefaultExecutionOrder(数值)]`，如 `[DefaultExecutionOrder(-100)]`                                                         |
| **编辑器与代码优先级**  | **编辑器面板中的设置优先**，会覆盖代码中的 `DefaultExecutionOrder`                                                                        |
| **未设置优先级的脚本**  | 全部在**Default Time** 槽位内以**不确定顺序**执行                                                                                         |
| **可靠设计建议**        | - 将不依赖他人的初始化放`Awake`，依赖他人的放 `Start`<br>- 使用集中式管理器手动控制初始化顺序<br>- 避免过度依赖脚本执行顺序来修正代码设计 |

整体阶段：
分阶段，统一执行。先执行所有脚本的Awake，然后再是OnEnable。逐次顺序执行。

同一阶段：
默认是随机执行。
设置优先级的情况下则是数字小的先执行。

同一脚本的不同实例：无法确定顺序

## 向量

### 定义

标量：只有数值
向量：有数值，也有方向。
向量的模：向量的大小
单位化/归一化：将向量转为单位向量的过程

### 运算

加法：A(x1,y2) + B(x2,y2) = (x1+x2,y1+y2)。几何意义上就是两个向量构建的平行四边形的对角线。
减法：A(x1,y2) - B(x2,y2) = (x1-x2,y1-y2)。几何意义上就是A的终点到B的终点的连线(平行四边形短的对角线)
乘法：A(x1,y2) * 2 = (x1 * 2,y1 * 2)

#### 一、点乘的数学定义

**点乘（Dot Product）** 是线性代数中两个向量之间的一种运算，结果是一个**标量**（一个数字）。

* **公式**（分量形式）：
  对于向量 `A = (x1, y1, z1)` 和 `B = (x2, y2, z2)`
  `A · B = x1*x2 + y1*y2 + z1*z2`
* **公式**（几何形式）：
  `A · B = |A| * |B| * cosθ`
  其中 `|A|`、`|B|` 是向量的长度（模），`θ` 是 A 与 B 之间的夹角（0\~π）。

#### 二、几何意义 —— 方向关系的“标尺”

从 `cosθ` 的符号和大小可以直观判断两个向量的方向关系：


| `A·B` 的值 | 对应的几何关系             | 夹角 θ            |
| ----------- | -------------------------- | ------------------ |
| **正数**    | 方向大致相同（指向同一侧） | 0° ≤ θ < 90°   |
| **0**       | 互相**垂直**（正交）       | θ = 90°          |
| **负数**    | 方向大致相反（指向相反侧） | 90° < θ ≤ 180° |
| `           | A                          |                    |
| `-          | A                          |                    |

> 当 A 和 B 都是**单位向量**（长度为1）时，`A·B = cosθ`，直接得到夹角的余弦值。

#### 三、在图形编程中的核心意义

点乘在 3D 图形学、游戏开发（Unity/Unreal）、着色器编程中无处不在，主要用来判断方向、角度和投影。

##### 🔹 1. 光照计算（漫反射光照 - Lambert）

* **公式**：`diffuse = max(0, normal · lightDir)`
* 用法：表面法线 `normal` 点乘 光源方向 `lightDir`。结果 >0 表示表面朝向光源，结果越大光照越强；结果为 0 或负值则无光。
* 这是所有实时渲染的基础。

##### 🔹 2. 判断物体的前后关系（视锥/视线）

* 用法：摄像机前向向量 `cameraForward` 点乘 从摄像机指向物体的向量 `dirToObject`。
  * 结果 >0 → 物体在摄像机**前方**。
  * 结果 <0 → 物体在**后方**（可能被剔除）。
* 类似用途：AI 视野检测（前方扇形范围）。

##### 🔹 3. 计算两个向量之间的夹角

* 用法：`θ = arccos( (A·B) / (|A||B|) )`
* 用途：计算转向角度、子弹命中角度、判断是否朝向目标等。

##### 🔹 4. 计算投影长度（投影）

* 用法：`projLength = A · (B / |B|)` 或直接用单位向量。
* 用途：例如将一个点的速度分解到某个轴的方向上。

##### 🔹 5. 镜面高光（Blinn-Phong 简化）

* 用法：半角向量 `H` 点乘法线 `N`，结果用于控制高光强度。

##### 🔹 6. 判定方向相似度

* 用法：例如 Steering Behavior 中，期望方向点乘当前方向，结果越接近 1 表示越一致。

#### 一、叉乘的数学定义

**叉乘（Cross Product）** 是只在三维空间中定义的向量运算，结果是一个**新的向量**，该向量垂直于原来的两个输入向量。

* **公式**（分量形式）：
  对于向量 `A = (x1, y1, z1)` 和 `B = (x2, y2, z2)`
  **text**

  ```
  A × B = ( y1*z2 - z1*y2 ,
            z1*x2 - x1*z2 ,
            x1*y2 - y1*x2 )
  ```
* **公式**（几何形式）：
  `|A × B| = |A| * |B| * sinθ`
  其中 `θ` 是 A 与 B 之间的夹角（0\~π）。

#### 二、几何意义 —— 生成垂直向量与有向面积

叉乘的结果向量 `C = A × B` 具有以下性质：


| 属性         | 含义                                                                                                                 |
| ------------ | -------------------------------------------------------------------------------------------------------------------- |
| **方向**     | 垂直于`A` 和 `B` 所张成的平面，遵循**右手定则**：<br/>四指从 `A` 弯向 `B`（角度 < 180°），拇指方向即为 `C` 的方向。 |
| **长度**     | `                                                                                                                    |
| **交换律**   | `A × B = - (B × A)`，叉乘反交换。                                                                                  |
| **特殊情况** | 若`A` 与 `B` 平行（`sinθ = 0`），则 `A × B = 0`（零向量）。                                                        |

> 右手定则图示：
>
> * 右手坐标系（Unity 世界坐标）：X右，Y上，Z前。
> * `X × Y = Z`，`Y × Z = X`，`Z × X = Y`。

#### 三、在图形编程中的核心意义

叉乘在 3D 图形学、游戏开发、物理引擎中主要用于**计算方向、法线、旋转轴和力矩**。

##### 🔹 1. 计算垂直向量（法线）

* **用法**：已知一个三角形的两条边向量 `AB` 和 `AC`，通过 `AB × AC`（或 `AC × AB`，取决于朝向）得到该三角形的**法线向量**。
* **应用**：生成网格的法线数据，用于光照计算；计算平面方程。

##### 🔹 2. 构建坐标系（基向量）

* **用法**：已知一个方向（例如前向 `forward`）和另一个参考方向（例如世界向上 `up`），通过叉乘补齐完整的局部坐标系的三个轴。
  * `right = up × forward`（归一化后）
  * `trueUp = forward × right`
* **典型场景**：摄像机跟随、导弹制导、`Transform.LookAt` 的内部实现。

##### 🔹 3. 判断方向性（左/右）

* **用法**：对于 `forward` 和 `toTarget`，计算 `cross = forward × toTarget`，然后检查 `cross.y` 的符号（在水平面上）：
  * `cross.y > 0` → 目标在**右侧**。
  * `cross.y < 0` → 目标在**左侧**。
* **应用**：绕 Y 轴旋转的转向判断、UI 箭头指示器。

##### 🔹 4. 计算力矩与角速度（物理）

* **用法**：力矩 `τ = r × F`，其中 `r` 是力臂向量，`F` 是力向量。叉乘结果的大小等于力矩大小，方向是旋转轴方向。
* **应用**：刚体受力产生旋转的物理模拟。

##### 🔹 5. 计算旋转轴（从一个向量旋转到另一个）

* **用法**：旋转轴 = `A × B`（归一化）。`A` 旋转到 `B` 的最短路径旋转轴垂直于两者平面。
* **应用**：`Quaternion.FromToRotation` 的原理，实现物体转向动画。

##### 🔹 6. 判断点是否在三角形内部（重心坐标辅助）

* 通过边缘叉乘符号一致性判断。

#### 插值

##### 一、插值的数学定义

插值的基本形式：给定起点 A**A** 和终点 B**B**，以及一个插值因子 t**t**（通常 0≤t≤1**0**≤**t**≤**1**），计算中间值 P**P**。

P=f(A,B,t)**P**=**f**(**A**,**B**,**t**)当 t=0**t**=**0** 时 P=A**P**=**A**；t=1**t**=**1** 时 P=B**P**=**B**；t=0.5**t**=**0.5** 时 P**P** 在中间位置。

最常见的插值类型是**线性插值（Linear Interpolation, Lerp）**：

* **标量（浮点数）**：
  Lerp(a,b,t)=a+(b−a)×t=(1−t)⋅a+t⋅b**Lerp**(**a**,**b**,**t**)**=**a**+**(**b**−**a**)**×**t**=**(**1**−**t**)**⋅**a**+**t**⋅**b
* **向量**：对每个分量分别线性插值。
  Lerp(A⃗,B⃗,t)=A⃗+(B⃗−A⃗)×t**Lerp**(**A**,**B**,**t**)**=**A**+**(**B**−**A**)**×**t
* **颜色**（RGBA）：对红、绿、蓝、透明度各通道分别线性插值。

##### 二、图形编程中的常见插值类型


| 类型               | 英文         | 数学特征                | 典型用途                               |
| ------------------ | ------------ | ----------------------- | -------------------------------------- |
| **线性插值**       | Lerp         | 匀速直线运动            | 简单移动、褪色、进度条                 |
| **球面线性插值**   | Slerp        | 沿球面最短弧插值        | 旋转（四元数）、摄像机朝向             |
| **平滑插值**       | SmoothStep   | 曲线先快后慢再快（S形） | 自然淡入淡出、摄像机缓动               |
| **三次贝塞尔插值** | Cubic Bezier | 自定义曲线形状          | 动画曲线、材质编辑器的动画曲线         |
| **指数/平滑阻尼**  | SmoothDamp   | 近似物理弹簧阻尼        | 平滑跟随、摄像机平滑跟随（Unity 内置） |

> **注**：对于四元数（Quaternion）旋转，必须用 **Slerp** 而非 Lerp，否则旋转路径会扭曲。

##### 三、在游戏开发中的核心应用

###### 🔹 1. 物体平滑移动

不用 `transform.position = targetPosition`（瞬移），而用 Lerp 每帧靠近一小段，产生跟随或缓动效果。

**csharp**

```
transform.position = Vector3.Lerp(transform.position, targetPosition, Time.deltaTime * speed);
```

###### 🔹 2. 颜色渐变

**csharp**

```
renderer.material.color = Color.Lerp(startColor, endColor, t);
```

###### 🔹 3. 摄像机平滑跟随

使用 `Vector3.SmoothDamp` 或 `Mathf.SmoothDamp` 实现带惯性效果的跟随，避免突兀。

###### 🔹 4. 旋转插值（四元数）

**csharp**

```
transform.rotation = Quaternion.Slerp(transform.rotation, targetRotation, Time.deltaTime * rotateSpeed);
```

###### 🔹 5. 动画混合

角色动画中，根据速度参数在 Idle 和 Run 动画之间插值权重；或使用 `Mathf.SmoothStep` 控制 UI 面板的显示/隐藏动画。

###### 🔹 6. 纹理采样与抗锯齿

GPU 在采样纹理时，根据 UV 坐标在两个相邻像素颜色之间做双线性插值（2D Lerp），使图像放大后不平滑缺失。
